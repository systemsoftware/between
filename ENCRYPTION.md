# Between App Encryption Documentation

The Between app ensures privacy by utilizing End-to-End Encryption (E2EE) for all messages sent over its peer-to-peer WebRTC data channels. The encryption is implemented using Apple's `CryptoKit` and `Security` frameworks.

## Key Generation and Storage

The app utilizes Elliptic Curve Cryptography (ECC) for key agreement, specifically the **NIST P-256** curve (`P256.KeyAgreement.PrivateKey`).

1.  **Key Generation**: When the app initializes, it checks if a private key already exists in the device's Secure Enclave/Keychain. If not, it generates a new random P-256 key pair (`kSecAttrKeyTypeECSECPrimeRandom`).
2.  **Keychain Storage**: The private key is persistently and securely stored in the iOS Keychain under the app group identifier. This ensures the key is protected by the device's hardware security and persists across app launches.
3.  **Public Key Exchange**: The public key is derived from the private key and shared with the peer during the initial WebRTC handshake (or manually out of band via the connection ID). The public key is represented as a Base64 encoded string of its raw representation.

## Encryption Process

When a user sends a message (text or image) to a peer, the following steps occur to encrypt the payload:

1.  **Peer Key Verification**: The app parses the target peer's Base64-encoded public key into a `P256.KeyAgreement.PublicKey`.
2.  **ECDH Key Agreement**: Using Elliptic-Curve Diffie-Hellman (ECDH), the app combines its local private key with the peer's public key to compute a **Shared Secret**.
3.  **Key Derivation**: The shared secret is passed through a Hash-based Message Authentication Code Key Derivation Function (HKDF) to generate a symmetric key suitable for encryption.
    *   **Hash Function**: SHA-256
    *   **Salt**: Empty (Data())
    *   **Shared Info**: `"P2P-Secure-Channel-Context"` (UTF-8 encoded)
    *   **Output Length**: 32 bytes (256 bits)
4.  **AES-GCM Encryption**: The derived symmetric key is used to encrypt the plaintext message using **AES-GCM** (Advanced Encryption Standard in Galois/Counter Mode). AES-GCM provides both confidentiality and authenticated encryption, ensuring the data cannot be read or tampered with.
5.  **Payload Bundling**: The resulting cipher, along with the nonce and authentication tag generated by AES-GCM, are combined into a single payload (`SealedBox.combined`).
6.  **Transmission**: The combined payload is Base64 encoded and sent over the WebRTC data channel.

## Decryption Process

When a client receives an encrypted message bundle:

1.  **Payload Decoding**: The incoming Base64 string is decoded into raw data and parsed into an `AES.GCM.SealedBox` to extract the cipher, nonce, and authentication tag.
2.  **Key Derivation**: The receiving client performs the same ECDH and HKDF process using its own local private key and the sender's public key to arrive at the exact same 32-byte symmetric key.
3.  **AES-GCM Decryption**: The app attempts to open the `SealedBox` using the derived symmetric key.
    *   If the authentication tag verifies successfully, the plaintext is returned.
    *   If the message was tampered with in transit, or if the keys do not match, the authentication fails, and the message is discarded.

## Security Guarantees

*   **End-to-End Confidentiality**: Because the symmetric encryption key is derived from a shared secret that only the two communicating peers can calculate, neither the signaling server nor any intermediate network node can decrypt the chat messages.
*   **Message Integrity**: AES-GCM provides authenticated encryption. Any modification to the encrypted payload in transit will result in a decryption failure on the receiving end.
*   **Secure Storage**: Private keys never leave the device and are protected by the iOS Keychain.
